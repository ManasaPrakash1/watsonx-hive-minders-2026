Alertes proactives

🔴 CRITIQUE — Mots de passe stockés et comparés en clair
Fichiers : UserServices.java:58, AdminServices.java:55

Impact métier : Si la base de données est compromise — par une fuite, une sauvegarde mal sécurisée ou une intrusion — tous les mots de passe des clients et administrateurs sont directement lisibles. Un attaquant peut se connecter immédiatement à n'importe quel compte sans déchiffrement. Le RGPD impose la protection des données personnelles : une telle violation peut entraîner une amende jusqu'à 4 % du chiffre d'affaires annuel mondial, sans compter le préjudice de réputation.
🔴 CRITIQUE — Mot de passe admin codé en dur dans le code source
Fichier : DataLoader.java:54 — "admin123" visible dans le dépôt Git

Impact métier : Tout développeur ayant accès au code source (y compris via des fuites GitHub) connaît le mot de passe administrateur par défaut. En production PostgreSQL, ce compte persiste et donne accès à l'intégralité du back-office : gestion des utilisateurs, commandes, catalogue produit.
🔴 CRITIQUE — Mots de passe exposés dans les méthodes toString()
Fichiers : AdminLogin.java:24, UserLogin.java:24

Impact métier : Toute journalisation des objets de connexion (en debug, dans les logs d'exception ou les traces APM) imprime le mot de passe en clair dans les fichiers de log. Ces fichiers sont souvent moins protégés que la base de données.
🔴 CRITIQUE — Placeholder OAUTH_USER codé en dur pour les utilisateurs Google
Fichier : OAuth2Controller.java:34

Impact métier : Tous les utilisateurs Google reçoivent le même mot de passe factice "OAUTH_USER". Si l'authentification par formulaire est activée sur les mêmes comptes, n'importe qui connaissant cet email peut se connecter à la place d'un utilisateur Google avec ce mot de passe.
🟠 URGENT — CSRF désactivé globalement
Fichier : SecurityConfig.java:21

Impact métier : Un attaquant peut créer une page web tierce qui déclenche silencieusement des actions au nom d'un utilisateur connecté (passer une commande, supprimer un compte, modifier des produits). Cette vulnérabilité est exploitable sans aucun accès au serveur.
🟠 URGENT — BCryptPasswordEncoder déclaré mais jamais câblé
Fichier : SecurityConfig.java:13 — bean déclaré, jamais injecté

Impact métier : L'investissement fait dans la déclaration du hachage est nul. Les développeurs suivants pourraient supposer que le hachage est actif alors qu'il ne l'est pas.
🟠 URGENT — 9 mutations via @GetMapping (delete, update)
Fichiers : AdminController.java, UserController.java, ProductController.java

Impact métier : Un lien simple dans un email ou une image intégrée suffit à déclencher une suppression de compte ou de produit (CSRF + GET = vulnérabilité combinée). Les navigateurs pré-fetching peuvent aussi déclencher ces actions involontairement.
🟡 ATTENTION — 4 services sans @Transactional
Fichiers : AdminServices, UserServices, ProductServices, OrderServices

Impact métier : En cas d'erreur mi-chemin dans une opération composée (créer une commande + mettre à jour le stock), la base de données peut rester dans un état partiellement modifié. Cela peut provoquer des données corrompues visibles par les clients.
🟡 ATTENTION — Couverture de tests à 67% (cible 80%)

Les 17 classes de test couvrent les principaux flux mais il manque 13 points de pourcentage pour atteindre le niveau de confiance recommandé. Les scénarios d'erreur (ID inexistant, utilisateur OAuth déjà existant) sont les zones les moins couvertes.

Backlog de remédiation priorisé

#1 CRITIQUE Câbler BCryptPasswordEncoder pour les mots de passe utilisateurs
Effort : 2jÉconomie de dette : 9jROI : 4.5×Taux : 800€/j
Fichiers : UserServices.java, UserController.java, AdminServices.java, SecurityConfig.java
Injecter le bean PasswordEncoder déjà déclaré dans SecurityConfig dans les services. À l'enregistrement, appeler encoder.encode(rawPassword). À la connexion, remplacer .equals(password) par encoder.matches(rawPassword, storedHash). Attention : une migration de données est requise pour les comptes existants en PostgreSQL (re-hash ou reset forcé des mots de passe).
#2 CRITIQUE Supprimer les mots de passe des méthodes toString()
Effort : 0.5jÉconomie de dette : 3jROI : 6.0×Taux : 800€/j
Fichiers : AdminLogin.java, UserLogin.java, Admin.java, User.java
Retirer le champ password / upassword des méthodes toString(). Remplacer par "[PROTECTED]" ou omettre le champ. Action de 30 minutes avec impact sécurité immédiat.
#3 CRITIQUE Externaliser le mot de passe admin du code source
Effort : 0.5jÉconomie de dette : 2jROI : 4.0×Taux : 800€/j
Fichier : DataLoader.java
Lire le mot de passe admin depuis une variable d'environnement (@Value("${admin.default.password:admin123}")) ou un secret manager. Ne jamais hardcoder de credential en clair dans le code versionné. Mettre à jour aussi la valeur dans .env.example.
#4 CRITIQUE Corriger le placeholder OAUTH_USER dans OAuth2Controller
Effort : 0.5jÉconomie de dette : 2jROI : 4.0×Taux : 800€/j
Fichier : OAuth2Controller.java:34
Générer un UUID aléatoire (UUID.randomUUID().toString()) ou un hash salé à la place du literal "OAUTH_USER". Les utilisateurs OAuth n'ont pas besoin de mot de passe formulaire — leur authentification passe par Google, mais un mot de passe prévisible crée un vecteur d'attaque.
#5 URGENT Ajouter @Transactional aux méthodes de mutation des services
Effort : 1jÉconomie de dette : 8jROI : 8.0×Taux : 600€/j
Fichiers : AdminServices.java, UserServices.java, ProductServices.java, OrderServices.java
Annoter les méthodes addXxx(), updateXxx(), deleteXxx() avec @Transactional (import org.springframework.transaction.annotation.Transactional). Les méthodes read-only peuvent utiliser @Transactional(readOnly = true) pour optimiser les performances.
#6 URGENT Ré-activer CSRF ou sécuriser les formulaires Thymeleaf
Effort : 1.5jÉconomie de dette : 2jROI : 1.3×Taux : 800€/j
Fichier : SecurityConfig.java
Option A (recommandée) : ré-activer CSRF et ajouter th:action="@{/...}" sur tous les formulaires Thymeleaf (Spring Security injecte automatiquement le token). Option B : implémenter un Double Submit Cookie pattern si CSRF ne peut pas être réactivé.
#7 URGENT Convertir les mutations GET en POST/DELETE
Effort : 1jÉconomie de dette : 1jROI : 1.0×Taux : 600€/j
Fichiers : AdminController.java, UserController.java, ProductController.java
Remplacer @GetMapping("/deleteXxx/{id}") par @PostMapping ou @DeleteMapping. Mettre à jour les templates Thymeleaf correspondants pour utiliser des formulaires POST avec méthode hidden _method=DELETE et le filtre HiddenHttpMethodFilter.
#8 MOYEN Sécuriser Optional.get() avec orElseThrow()
Effort : 0.5jÉconomie de dette : 1.5jROI : 3.0×Taux : 500€/j
Fichiers : AdminServices.java:27, UserServices.java:28, ProductServices.java:31+39
Remplacer optional.get() par optional.orElseThrow(() -> new EntityNotFoundException("Entity not found: " + id)). Cela garantit un message d'erreur exploitable plutôt qu'une NoSuchElementException générique.
#9 MOYEN Refactoriser AdminController (God class)
Effort : 2jÉconomie de dette : 4jROI : 2.0×Taux : 600€/j
Fichier : AdminController.java (219 lignes, 17 méthodes, 4 services injectés)
Extraire les opérations produit vers ProductController (déjà existant), les opérations utilisateur vers UserController, les opérations commande vers un nouveau OrderController. Ne conserver dans AdminController que l'authentification admin et le dashboard.
#10 FAIBLE Remplacer @Component par @Service sur les services
Effort : 0.2jÉconomie de dette : 0.5jROI : 2.5×Taux : 500€/j
Fichiers : AdminServices.java, UserServices.java, ProductServices.java, OrderServices.java
Remplacer @Component par @Service. Fonctionnellement identique, mais améliore la lisibilité et respecte les conventions Spring (rend le rôle de chaque classe explicite pour les outils d'analyse statique et les nouveaux développeurs).
#11 FAIBLE Stocker un DTO en session plutôt que l'entité JPA User
Effort : 1jÉconomie de dette : 1jROI : 1.0×Taux : 500€/j
Fichiers : AdminController.java:62, OAuth2Controller.java:42
Créer un record UserSessionDto(int id, String name, String email) et stocker ce DTO en session à la place de l'entité JPA complète. Évite les LazyInitializationException et réduit la taille de la session HTTP sérialisée.
#12 FAIBLE Remplacer System.out.println() par un logger SLF4J
Effort : 0.5jÉconomie de dette : 0.5jROI : 1.0×Taux : 500€/j
Fichiers : DataLoader.java, SecurityConfig.java, OAuth2Controller.java, UserController.java, Exceptions.java
Utiliser private static final Logger log = LoggerFactory.getLogger(Xxx.class); puis log.info() / log.debug(). Les loggers SLF4J sont configurables par niveau, gèrent les sorties structurées, et n'impriment pas en production.
#13 FAIBLE Ajouter un stage SAST au pipeline CI
Effort : 0.5jÉconomie de dette : 1jROI : 2.0×Taux : 550€/j
Fichier : .github/workflows/trigger-onboarding-agent.yml
Ajouter une étape ./mvnw test (ou SpotBugs / OWASP Dependency Check) avant le build dans le pipeline GitHub Actions ou le buildspec.yml AWS. Actuellement, buildspec.yml compile avec -DskipTests — les tests ne sont jamais exécutés en CI.
#14 FAIBLE Augmenter la couverture de tests de 67% à 80%
Effort : 1.3jÉconomie de dette : 1.3jROI : 1.0×Taux : 450€/j
Priorité : cas d'erreur (ID inexistant dans les services), scénario OAuth avec utilisateur existant, comportement après logout, routes d'administration sans session. Les tests *RepositoryTest.java et *ServicesTest.java existent déjà — compléter les cas limites.
