🚨 Alertes actives

CritiqueMots de passe stockés et comparés en clair
Les mots de passe des utilisateurs et des administrateurs sont stockés en base de données sans aucun hachage, et comparés directement en texte clair lors de la connexion. En cas de fuite de la base de données — même accidentelle — l'ensemble des comptes utilisateurs est immédiatement compromis. Le bean BCryptPasswordEncoder est déjà déclaré dans le code mais n'est jamais utilisé : le correctif est à portée de main.
CritiqueMot de passe admin codé en dur dans le code source
Le mot de passe admin123 est écrit en clair dans le fichier DataLoader.java, versionné dans Git. Toute personne ayant accès au dépôt — y compris d'anciens collaborateurs — connaît les credentials d'administration de production. Ce type de secret ne doit jamais apparaître dans le code source.
CritiqueConsole H2 exposée publiquement sans authentification
La console d'administration de la base de données H2 est accessible à n'importe qui sur l'URL /h2-console sans aucune restriction d'accès. En environnement de développement c'est acceptable, mais si cette configuration est déployée en production, un attaquant peut lire, modifier ou supprimer l'intégralité des données de l'application.
Urgent9 mutations d'état via HTTP GET
Neuf opérations de modification ou suppression de données (utilisateurs, produits, admins) sont exposées via des requêtes GET au lieu de POST/PUT/DELETE. Cela signifie qu'un simple lien dans un email ou une image dans une page web peut déclencher une suppression de données à l'insu de l'utilisateur (attaque CSRF par GET), même si le CSRF est par ailleurs désactivé.
UrgentPlaceholders OAuth2 non remplacés dans application.properties
Les valeurs YOUR_CLIENT_ID et YOUR_CLIENT_SECRET sont versionnées dans application.properties. Si ce fichier est déployé tel quel, la configuration OAuth2 est non fonctionnelle. De plus, ce pattern encourage les développeurs à mettre de vrais secrets dans ce fichier, qui est tracké par Git.
AttentionCouverture de tests à 33% — cible 80% non atteinte
Seul un tiers du code est couvert par des tests automatisés. Les modules les plus risqués (authentification, gestion des commandes, logique de prix) ne sont pas testés. Tout refactoring dans ces zones est réalisé à l'aveugle, avec un risque élevé de régression silencieuse.
AttentionAucune transaction explicite (@Transactional)
Aucune méthode de service n'est annotée @Transactional. Les opérations impliquant plusieurs sauvegardes successives peuvent laisser la base dans un état incohérent si une erreur survient entre deux appels à save().
RésoluCouverture tests < 20% (v1.0 → v2.0)
La couverture est passée de 4% à 33% entre la v1.0 et la v2.0 grâce à l'ajout de 8 fichiers de test et 62 tests. Cette amélioration est maintenue en v3.0.

🎯 Backlog priorisé — Top 10 actions

#1[CRITIQUE] Activer le hachage des mots de passe
Effort : 1jÉconomie de dette : 8jROI : 8×
UserServices.java · AdminServices.java · SecurityConfig.java · DataLoader.java
Le bean BCryptPasswordEncoder est déjà déclaré dans SecurityConfig. Il suffit de l'injecter dans UserServices et AdminServices, d'encoder les mots de passe à la création (addUser, addAdmin) et d'utiliser passwordEncoder.matches() à la place de .equals() lors de la validation. Mettre à jour DataLoader pour encoder le mot de passe admin par défaut.
#2[CRITIQUE] Externaliser les secrets hors du code source
Effort : 0.5jÉconomie de dette : 5jROI : 10×
DataLoader.java · application.properties
Remplacer "admin123" dans DataLoader.java par une variable d'environnement (@Value("${admin.default.password}")). Déplacer les credentials OAuth2 dans un fichier application-local.properties exclu du Git (.gitignore).
#3[CRITIQUE] Restreindre l'accès à la console H2
Effort : 0.5jÉconomie de dette : 3jROI : 6×
SecurityConfig.java · application.properties
Conditionner l'accès à /h2-console/** au profil Spring dev uniquement via @Profile("dev") sur le bean de configuration, ou désactiver spring.h2.console.enabled dans les profils prod et staging.
#4[URGENT] Remplacer les GET de mutation par POST/DELETE
Effort : 2jÉconomie de dette : 4jROI : 2×
AdminController.java · UserController.java · ProductController.java
Convertir les 9 endpoints @GetMapping("/updating*") et @GetMapping("/delete*") en @PostMapping ou @DeleteMapping avec CSRF ré-activé. Mettre à jour les formulaires Thymeleaf correspondants avec method="post" et th:action.
#5[URGENT] Ajouter @Transactional sur les services
Effort : 0.5jÉconomie de dette : 2jROI : 4×
UserServices.java · AdminServices.java · ProductServices.java · OrderServices.java
Annoter les méthodes de service qui font plusieurs opérations DB avec @Transactional. Marquer les méthodes en lecture seule avec @Transactional(readOnly = true) pour optimiser les performances.
#6[ATTENTION] Remplacer la validation login user par une requête ciblée
Effort : 0.5jÉconomie de dette : 1.5jROI : 3×
UserServices.java · UserRepository.java
validateLoginCredentials() charge tous les utilisateurs en mémoire pour comparer. Remplacer par findUserByUemailAndUpassword(email, encodedPassword) dans UserRepository après activation du hachage (#1).
#7[ATTENTION] Augmenter la couverture de tests à 60%+
Effort : 5jÉconomie de dette : 5jROI : 1×
UserServices.java · AdminController.java · OrderServices.java
Prioriser les tests sur les flux critiques : login admin/user, placement de commande, calcul du total. Utiliser @SpringBootTest + MockMvc pour les controllers et @DataJpaTest pour les repositories.
#8[ATTENTION] Corriger le bug dans AdminServices.update()
Effort : 0.5jÉconomie de dette : 1jROI : 2×
AdminServices.java
La méthode update(Admin admin, int id) itère les admins pour trouver l'ID correspondant mais n'appelle pas admin.setAdminId(id) avant save(admin). L'admin mis à jour peut donc être créé avec un nouvel ID au lieu d'être mis à jour.
#9[QUALITÉ] Corriger OrderController (classe vide)
Effort : 1jÉconomie de dette : 0.5jROI : 0.5×
OrderController.java · AdminController.java
OrderController est une classe vide décorée de plusieurs imports non utilisés. La logique des commandes est dispersée dans AdminController. Extraire les handlers de commande dans OrderController pour respecter le principe de responsabilité unique.
#10[QUALITÉ] Migrer CrudRepository vers JpaRepository
Effort : 0.5jÉconomie de dette : 0.5jROI : 1×
UserRepository.java · AdminRepository.java · ProductRepository.java
Trois des quatre repositories étendent CrudRepository (qui retourne Iterable) au lieu de JpaRepository (qui retourne List). Cela force des casts explicites dans tous les services et bloque l'accès à la pagination et au tri.

📋 Note méthodologique

Cet audit a été réalisé par analyse statique manuelle du code source (grep, read_file) en mode fallback (Node.js non disponible). Le modèle de coût applique un taux journalier de 550 €/j. La dette est calculée selon une grille standardisée : finding critique = 3j, finding high = 1.5j, finding medium = 0.5j, écart de couverture = 1j par tranche de 10% sous 80%, code smell = 0.3j/fichier. Les CVE sont évaluées qualitativement (pas de base NVD interrogée en temps réel).
