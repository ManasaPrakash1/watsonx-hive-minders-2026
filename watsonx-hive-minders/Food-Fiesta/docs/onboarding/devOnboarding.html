1. Purpose

Food Fiesta is a full-stack dining management application that enables customers to browse a food menu, place orders, and track their order history. Administrators can manage the product catalog, user accounts, and view all orders through a dedicated back-office dashboard.

Business Domain

Online food ordering and restaurant management system with dual-role access control.

User Roles

  • Customer — Browse products, search for dishes, place orders, view order history
  • Administrator — Full CRUD operations on users, admins, products, and orders; access to analytics dashboard

Core Features

  • Public product catalog with search functionality
  • Customer self-registration and login (email/password or Google OAuth2)
  • Order placement with automatic total calculation
  • Admin dashboard for managing all entities (users, products, orders, admins)
  • Session-based authentication with Spring Security route protection
  • Interactive API documentation via Swagger UI
  • H2 in-memory database for local development, PostgreSQL support for production

3. Application Architecture

Architectural Layers

Food Fiesta follows a classic Spring Boot layered architecture:

  • Controllers (com.example.demo.controllers) — HTTP request handlers, route definitions, session management
  • Services (com.example.demo.services) — Business logic, validation, orchestration between repositories
  • Repositories (com.example.demo.repositories) — Data access layer, extends CrudRepository or JpaRepository
  • Entities (com.example.demo.entities) — JPA entities mapping to database tables
  • Config (com.example.demo.config) — Security configuration, data seeding, OpenAPI setup

Authentication Mechanism

Dual-track session-based authentication:

  • Spring Security handles route authorization only — it does NOT perform credential validation
  • Manual credential validation is done in AdminController:
    • AdminController.getAllData() validates admin login via AdminServices.validateAdminCredentials()
    • AdminController.userLogin() validates user login via UserServices.validateLoginCredentials()
  • Session storage:
    • Admin: session.setAttribute("loggedInAdmin", email) — stores email as String
    • User: session.setAttribute("loggedInUser", user) — stores full User JPA entity
  • Passwords are stored in plain text — BCryptPasswordEncoder bean exists in SecurityConfig but is not applied anywhere
  • OAuth2 Google login creates a User with placeholder values: upassword = "OAUTH_USER", unumber = 0L
  • CSRF is disabled globally in SecurityConfig

Data Model

Entity Table Name Key Fields Relationships
User customer_info u_id (PK), uemail, upassword, uname, unumber One-to-Many with Orders
Admin admin adminId (PK), adminEmail, adminPassword, adminName, adminNumber None
Product product_table pid (PK), pname, pprice, pdescription None
Orders order_info oId (PK), oName, oPrice, oQuantity, totalAmmout, orderDate Many-to-One with User
⚠️ Note: Orders.totalAmmout has an intentional typo (double "m") — this maps to the database column and must NOT be changed without a migration.

Request Lifecycle

  1. Browser sends HTTP request (Thymeleaf form submission or direct navigation)
  2. Spring Security checks if the route requires authentication (see SecurityConfig.securityFilterChain())
  3. Controller receives the request, validates session attributes if needed
  4. Service executes business logic, calls repositories
  5. Repository performs database operations via JPA/Hibernate
  6. Database (H2 or PostgreSQL) returns data
  7. Controller populates the model and returns a Thymeleaf template name or redirect
  8. Thymeleaf renders the HTML response

Non-Obvious Patterns

  • Services use @Component instead of @Service — This is intentional and works correctly
  • Repositories extend CrudRepository (except OrderRepository which extends JpaRepository) — findAll() returns Iterable, always cast to List
  • Update operations use @GetMapping instead of @PutMapping — This is the established pattern for all updating* routes
  • Orders uses GenerationType.AUTO while all other entities use IDENTITY — Can cause sequence conflicts on PostgreSQL
  • User.u_id field uses underscore — Getter/setter are getU_id() / setU_id()
  • Logic.countTotal(price, quantity) in com.example.demo.count is the only calculation utility — Use it for order totals, do not inline multiplication

5. CI/CD & Deployment

Continuous Integration — AWS CodeBuild

The project uses AWS CodeBuild for automated builds, configured via buildspec.yml:

ℹ️ Note: Tests are skipped in the CI build (-DskipTests). This is because tests require Spring context with valid OAuth2 credentials, which are not configured in the CI environment.

Docker — Containerization

Dockerfile Structure

The project uses a multi-stage Docker build:

Stage 1: Build

  • Base image: eclipse-temurin:21-jdk-alpine
  • Copies Maven wrapper and pom.xml
  • Downloads dependencies (cached layer)
  • Copies source code and builds with ./mvnw clean package -DskipTests

Stage 2: Run

  • Base image: eclipse-temurin:21-jre-alpine (minimal runtime)
  • Creates non-root user spring:spring for security
  • Copies JAR from build stage
  • Exposes port 8080
  • Runs via entrypoint.sh script

7. Known Gotchas & Contribution Guide

Known Gotchas

These are non-obvious patterns that would trip up a new developer. Do not "fix" these without team discussion:

⚠️ Intentional Typo: Orders.totalAmmout has a double "m" — this maps to the database column name. Changing it will break JPA column mapping.
⚠️ Non-Standard Annotation: All service classes use @Component instead of @Service. This is intentional and works correctly. Do not change it.
⚠️ Unsafe Pattern: AdminServices.update() and ProductServices.updateproduct() call Optional.get() without checking isPresent(). This throws NoSuchElementException if the ID doesn't exist. This is a known bug but not yet fixed.
⚠️ Wrong HTTP Verb: Update operations use @GetMapping instead of @PutMapping or @PostMapping. This is the established pattern for all updating* routes. Do not change it without updating all Thymeleaf forms.
⚠️ Hardcoded Credentials: Default admin password is admin123 (seeded in DataLoader). This is for local development only. Do not change it without updating team documentation.
⚠️ Session Storage: session.setAttribute("loggedInUser", user) stores a full JPA entity. If the session object becomes detached from the Hibernate session, lazy-loaded fields (orders) will throw LazyInitializationException.
⚠️ Plain-Text Passwords: Both admin and user passwords are stored in plain text. BCryptPasswordEncoder bean exists but is not applied anywhere in the current codebase.
⚠️ Scalability Issue: UserServices.validateLoginCredentials() loads all users from the database for comparison. This is not scalable. Should be replaced with a findUserByUemailAndUpassword query method.
⚠️ No Transactions: No @Transactional annotations anywhere. All service methods run without explicit transaction boundaries. Complex multi-save operations are at risk of partial commits.

Contribution Guide

Running Tests Locally

./mvnw test

On Windows:

.\mvnw.cmd test
ℹ️ Test Pass Rate: All existing tests must pass before pushing. Currently, there is only one test file (FoodFrenzyApplicationTests) that checks if the Spring context loads.

Branching Convention

The project does not enforce a specific branching strategy in the CI configuration. Follow your team's established Git workflow (e.g., feature branches merged to main via pull request, or direct push to main).

Running the App Before Pushing

./mvnw spring-boot:run

Verify that:

  • The application starts without errors
  • You can access http://localhost:8080/
  • Login functionality works (admin and user)

CI Checks

On push, AWS CodeBuild will automatically:

  • Verify Maven and Java versions
  • Run mvn clean package -DskipTests
  • Produce a JAR artifact in output/
✅ Testing Reminder: If you add a new service method, add a matching test in the corresponding *Test.java file (e.g., AdminServicesTest, UserServicesTest).

Code Style

  • Follow existing naming conventions (e.g., u_id, totalAmmout)
  • Use @Component for service classes (not @Service)
  • Cast findAll() results to List explicitly
  • Use Logic.countTotal(price, quantity) for order calculations
  • Add Swagger annotations (@Operation, @Tag) to all new controller methods
