1 — Purpose

Food Fiesta is a full-stack dining management web application built with Spring Boot 3.4.2 and Java 21. It provides a premium online food-ordering experience for customers while giving restaurant administrators a complete back-office console to manage the menu, users, orders, and staff accounts.

User Roles

RoleAccessHow to authenticate
AdminFull back-office: manage products, users, orders, other adminsCustom form login (/login) validated against the admin table; session-based
CustomerBrowse menu, place orders, view order historyCustom form login (/userLogin) or Google OAuth2; session-based
AnonymousHome, Products, About, Locate Us — read onlyNo authentication needed

Core Features

  • Thymeleaf SSR UI with custom CSS and JavaScript
  • Role-based dual login system (admin + customer) with optional Google OAuth2
  • Product (food item) catalog with CRUD management from the admin panel
  • Customer registration and self-service account management
  • Order placement with automatic total calculation and order history
  • Spring Data JPA persistence (H2 in-memory for local dev, PostgreSQL for production)
  • Swagger / OpenAPI 3 documentation at /swagger-ui/index.html
  • Health-check REST endpoint at /api/health

Authentication Mechanism

Food Fiesta uses a hybrid authentication model:

  • Custom session-based login for admins — /adminLogin (POST) manually checks email + plaintext password against the admin table via AdminServices.validateAdminCredentials(). On success, the admin email is stored in HttpSession as "loggedInAdmin". Spring Security is not involved in this flow.
  • Custom session-based login for customers — /userLogin (POST) manually checks email + plaintext password via UserServices.validateLoginCredentials(). On success, the full User JPA entity is stored in HttpSession as "loggedInUser".
  • Google OAuth2 for customers — Spring Security's OAuth2 client handles the Google flow; on success OAuth2Controller.oauth2Success() creates or retrieves a local User and puts it in the session like a normal login.
  • Spring Security is configured to permit a fixed list of public routes and require authentication for everything else. However, the admin and user dashboards rely on manual session checks inside controller methods, not on Spring Security's access rules.
  • CSRF is disabled — intentional (simplifies Thymeleaf form handling), but is a known security gap.
  • Passwords for customers are stored and compared in plaintext — BCryptPasswordEncoder is wired in the bean but never applied in registration or login flows.

Request Lifecycle

Browser → Spring Security filter chain (permit public / block protected) → Controller method → Service method → Repository (Spring Data JPA / Hibernate) → H2 or PostgreSQL → entity returned → Thymeleaf template rendered → HTTP response sent to browser.

Seeded Data on Startup

DataLoader (a @Configuration + CommandLineRunner bean) seeds 5 products and 1 default admin account if the tables are empty. It runs on every startup since H2 uses create-drop.

7 — Known Gotchas & Contribution Guide

Known Gotchas

⚠️ Passwords are stored and compared in plaintext for customers. UserServices.validateLoginCredentials() does a plain .equals(password) comparison. BCryptPasswordEncoder is declared as a bean in SecurityConfig but is never injected into the registration or login flows. Do not "fix" this silently — it would break all existing user accounts in production PostgreSQL.
⚠️ Admin passwords are also plaintext. AdminServices.validateAdminCredentials() uses .equals(password). The default seeded admin (admin123) is stored with no hashing.
⚠️ Optional.get() without guard — potential NoSuchElementException. AdminServices.getAdmin(id), UserServices.getUser(id), and ProductServices.getProduct(id) all call optional.get() directly. If an ID that doesn't exist is passed (e.g. via a crafted URL), the application will throw an unguarded exception. The project catches this globally in Exceptions.java, but the UX is a generic error page.
⚠️ All service classes use @Component instead of @Service. AdminServices, UserServices, ProductServices, and OrderServices are all annotated @Component. This works in Spring but goes against convention. Do not rename the annotation without a team discussion — it is intentional in this codebase.
⚠️ Full JPA entity stored in HttpSession as loggedInUser. The User entity (with its lazily loaded orders list) is stored directly in the HTTP session. This can cause LazyInitializationException or stale-data issues after the Hibernate session closes. Do not replace this with a DTO without testing all session-dependent flows.
⚠️ Mutation endpoints mapped to @GetMapping. /updatingAdmin/{id}, /updatingUser/{id}, /updatingProduct/{productId}, /deleteAdmin/{id}, /deleteUser/{id}, /deleteProduct/{productId} all use GET — not PUT/DELETE. This is how the Thymeleaf forms are wired. Do not change the HTTP verbs without updating all templates.
ℹ️ totalAmmout has a double letter — this is intentional. The field is spelled totalAmmout (double m, missing n) in both the Orders entity and the order_info DB column. The getters/setters match. Never "fix" the spelling — it would require a DB migration.
ℹ️ CSRF is disabled project-wide. SecurityConfig calls csrf(csrf -> csrf.disable()). This is intentional to keep Thymeleaf form submissions simple. Do not re-enable without adding CSRF tokens to all forms.
ℹ️ The main class is named FoodFrenzyApplication, not FoodFiestaApplication. The artifact name is "Food Fiesta" but the main class kept its original name. Don't rename it — it changes the JAR entry point.
